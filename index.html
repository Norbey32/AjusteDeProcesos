Metodología Seleccionada

Tras analizar el componente formativo “Aplicación de pruebas de software” y el proyecto SistemaVentasDjango (un sistema de ventas en Django con modelos para productos, ventas y usuarios), se reconoce que la metodología seleccionada es Scrum. Este proyecto parece haber sido desarrollado de manera iterativa, pero sin evidencia explícita de sprints o ceremonias en el repositorio (e.g., commits sin referencias a retrospectives). Scrum se adapta bien a proyectos Django, promoviendo entregas incrementales de funcionalidades como gestión de inventarios o reportes de ventas.

Propuestas de Cambios en la Secuencia de Procesos

Basado en el comportamiento observado en el proyecto SistemaVentasDjango (e.g., código en `views.py` sin pruebas unitarias, y estructura de apps que podría beneficiarse de refactorización), se proponen ajustes en la secuencia de ejecución de procesos de Scrum para optimizar el desarrollo.

Cambios Operativos:

  • Secuencia de Ceremonias: Introducir sprints de 2 semanas para el proyecto, con Sprint Planning enfocada en features como "Agregar autenticación de usuarios" (basado en `models.py`). Ajustar la secuencia estándar para incluir Daily Scrums virtuales, dado que el repo sugiere desarrollo individual.
  • Roles y Responsabilidades: Designar un Product Owner para priorizar backlog (e.g., mejoras en reportes de ventas), y un Scrum Master para facilitar reuniones, reduciendo la dependencia de commits ad-hoc.

Cambios Técnicos:

  • Integración Continua (CI/CD): Implementar GitHub Actions en el repo para ejecutar pruebas Django (e.g., `python manage.py test`) al final de cada sprint, ajustando la secuencia de desarrollo (Desarrollo en `views.py` → Pruebas → Despliegue en Heroku).
  • Pruebas de Software: Adoptar TDD para escribir pruebas en `tests.py` antes de modificar modelos (e.g., probar lógica de ventas). Integrar code reviews en Pull Requests para validar cambios en `settings.py`.

Aplicación de Cambios:

Estos cambios se aplicarán en el próximo sprint del proyecto: operativamente, mediante commits documentados; técnicamente, agregando un archivo `requirements.txt` para dependencias y configurando un pipeline CI.

Buenas Prácticas de Calidad

Para el proyecto SistemaVentasDjango, se seleccionan prácticas alineadas con Scrum y el Agile Manifesto: Uso de Django ORM para modelos limpios (e.g., en `models.py`), métricas de cobertura de código con tools como Coverage.py, y adherencia a "Done" definiendo criterios como "venta registrada con validación". Estas prácticas mejoran la calidad, reduciendo bugs en funcionalidades como reportes, mediante inspección iterativa.

Documentación de los Cambios

Los cambios se documentan en el repositorio GitHub del proyecto: Crear un archivo `CHANGELOG.md` con versiones de sprints, y usar la wiki para diagramas de flujo (e.g., secuencia de ventas). Incluir un `README.md` actualizado con instrucciones de instalación y referencias a Scrum. Esta documentación facilita colaboraciones y auditorías futuras.
